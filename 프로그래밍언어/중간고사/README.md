# 1장 프로그래밍 언어론

## 언어 평가 기준 4가지 및 평가 요소

### 1. 가독성 (Readability)

정의 : 프로그램을 읽고 이해하기 쉬운 정도
<br>
평가 요소
- 전반적 단순성 : 기능 및 구성 최소화, 기능 중복과 과도한 연산자 오버로딩이 없어야 함
- 직교성 : 적은 수의 기본 구성 요소를 일관되게 조합할 수 있어야 하며, 가 능한 조합은 예외 없이 동작해야 함
- 데이터 타입 : 사전 정의된 데이터 타입이 충분히 제공될수록 사용자의 의도를 명확히 표현할 수 있음
- 구문 설계 : 식별자 형태, 복합문 구성 방식, 의미를 드러내는 키워드 사용이 이해도에 직접적 인 영향을 미침

<br>

### 2. 작성 용이성 (Writability)

정의 : 해당 언어로 프로그램을 작성하기 쉬운 정도

<br>

평가 요소
- 단순성과 직교성 : 적은 수의 기본 구조와 조합 규칙으로 복잡한 프로그램을 구성할 수 있어야 함
- 추상화 지원 : 복잡한 구조나 연산을 정의하고, 세부 구현을 드러내지 않고 사용할 수 있어야 함
- 표현력 : 간결한 코드로 많은 의미를 전달할 수 있어야 하며 연산자와 사전 정의 함수가 이를 지원함

<br>

### 3. 신뢰성 (Reliability)

정의 : 프로그램이 명세에 따라 정확하게 동작하는 정도

<br>

평가 요소
• 타입 검사 : 컴파일 시점 또는 실행 시점에 타입 오류를 검출하여 잘못된 연산을 사전에 방지
• 예외 처리 : 실행 중 발생하는 오류를 감지하고 처리할 수 있도록 지원
• 앨리어싱 : 같은 메모리 위치에 여러 참조 경로가 존재하는 상황, 앨리어싱이 많을수록 프로그램의 동작을 예측하기 어려움


<br>


### 4. 비용 (Cost)
설계부터 유지보수까지 전체 생명주기 관점에서 평가

|비용 유형|설명|
|:---:|:---|
|교육 비용|언어를 익히는데 드는 시간과 노력|
|개발 비용|같은 기능 구현에 드는 시간과 노력|
|실행 비용|컴파일 시간, 실행 시간, 메모리 사용량|
|신뢰성 비용|오류 발생 시 수정에 드는 시간과 비용|
|유지보수 비용|배포 후 버그 수정, 기능 개선, 코드 이해에 드는 비용. 전체 생명주기에서 비중이 가장 큰 경우가 많음|

## 언어 범주 4가지의 특징과 대표 언어

|범주|특징|대표 언어|
|:---:|:---|:---:|
|명령형|변수, 할당문, 반복문 중심. 메모리 상태 변경|C, Java, C++|
|함수형|함수 적용으로 계산. 상태 변경 최소화|Lisp, Scheme, ML, F#|
|논리형|규칙 기반 추론. 사실과 규칙 선언|Prolog|
|혼합형|마크업·질의 언어에 제어·변환 기능을 결 합|XSLT, JSTL|

|JIT|중간 코드 + 실행 시 필요한 부분만 기계어 변환|빠름|높음|Java, .NET|

# 3장 구문과 의미론

## 구문 기술 기본 용어

|용어|정의|예시|
|:---:|:---|:---:|
|문장|정해진 알파벳으로 만들어진 문자열|—|
|언어|문장들의 집합|—|
|렉심|언어에서 가장 작은 구문 단위|* , sum , begin|
|토큰|렉심들을 분류한 범주|식별자|

<br>
<br>

## 리스트의 재귀적 정의 (종료 규칙, 재귀 규칙)

- 종료 규칙(Base Case) : 식별자 하나만 있는 경우. 리스트 종료
- 재귀 규칙(Recursive Case) : 쉼표 뒤에 식별자 리스트가 다시 이어지는 형태. 리스트 확장

<br>

두 규칙은 항상 함께 필요하다. 재귀 규칙만 있으면 종료할 수 없고, 종료 규칙만 있으면 확장할 수 없음

<br>
<br>

## 속성 문법의 속성 종류(합성, 상속, 고유)

|속성|방향|설명|
|:---:|:---:|:---|
|합성 속성|아래 → 위|자식 노드의 속성 값으로 부모 노드의 속성 계산|
|상속 속성|위 → 아래|부모 노드의 속성으로 자식 노드의 속성 계산|
|고유 속성|—|잎 노드에 처음부터 주어지는 초기 속성 값|

<br>
<br>

## 의미론 3가지 비교 (조작적, 표시적, 공리적)

|항목|조작적 의미론|표시적 의미론|공리적 의미론|
|:---:|:---|:---|:---|
|의미 정의 방식|프로그램 실행|수학적 함수|논리적 단언|
|기반 이론|가상 머신 실행 모델|재귀 함수 이론|술어 논리|
|주요 용도|프로그램 동작의 직관적 설명|언어 설계, 컴파일러 생성, 정확성 증명|프로그램의 형식적 검증|
|추상화 수준|낮음 (구체적 실행 과정)|높음 (수학적 추상화)|중간 (논리적 추상화)|

# 4장 어휘 분석과 구문 분석

<br>

## 어휘 분석기/파서의 기반 문법 및 수학적 모델

### 하위 수준 : 어휘 분석기 (Lexical Analyzer)
- 문법 : 정규 문법 기반
- 수학적 모델 : 유한 오토마톤 (Finite Automaton)

<br>

### 상위 수준 : 구문 분석기 / 파서 (Parser)
- 문법 : 문맥 자유 문법(BNF) 기반
- 수학적 모델 : 푸시다운 오토마톤 (Push-Down Automaton)

<br>
<br>

## 어휘 분석기 구축 방법 3가지와 유틸리티 함수 역할

- 토큰 형식 기술
  - 소프트웨어 도구로 테이블 구동 분석기 자동 생성
- 상태 다이어그램 설계
  - 프로그램으로 직접 구현
- 상태 다이어그램 설계
  - 전이 테이블 기반 수작업 구현
 
### 유틸리티 함수

|함수|역할|
|:---:|:---|
|getChar|다음 문자 읽기, nextChar에 저장, charClass에 문자 클래스 저장|
|addChar|nextChar의 문자를 렉심(lexeme)에 추가|
|lookup|lexeme이 예약어인지 판별, 토큰 코드 반환|

<br>
<br>

## 예약어 인식 방법

예약어 : 식별자와 동일한 패턴
- 별도 다이어그램 불필요
- 식별자로 먼저 인식
  - 테이블 조회(Table Lookup)로 예약어 여부 판별
 
<br>
<br>

## 하향식 파서와 상향식 파서 비교

|구분|하향식 (Top-Down)|상향식 (Bottom-Up)|
|:---:|:---|:---|
|방향|루트 → 리프|리프 → 루트|
|유도|레프트모스트 유도||라이트모스트 유도의 역순|
|트리 순회|전위 순서 (Preorder)|-|
|대표 알고리즘|재귀 하강(RD), LL 파서|LR 파서|
|구현 방식|코드 기반(RD), 테이블 기반(LL)|테이블 기반|
